حتماً. اگر قرار است **برای اولین بار** StatefulSet را تدریس کنید، مهم‌ترین نکته این است که **از خود StatefulSet شروع نکنید**. اول باید دانشجو احساس کند چرا Deployment برای بعضی سرویس‌ها کافی نیست. وقتی مشکل را درک کند، خودش به StatefulSet می‌رسد.

من اگر این مبحث را تدریس می‌کردم، تقریباً به این شکل پیش می‌رفتم.

---

# StatefulSet در Kubernetes

## مقدمه

تا اینجا یاد گرفتیم که **Deployment** بهترین انتخاب برای اجرای اکثر برنامه‌های Stateless است.

برای مثال:

* یک وب‌سایت
* یک REST API
* یک Backend Service
* یک Frontend
* یک Microservice

در تمام این موارد، اگر یکی از Podها حذف شود، Kubernetes به سادگی یک Pod جدید ایجاد می‌کند.

از دید برنامه، تفاوتی وجود ندارد که درخواست توسط Pod شماره ۱ پاسخ داده شود یا Pod شماره ۲.

چون این برنامه‌ها **State** یا وضعیت دائمی داخل خود Pod نگه نمی‌دارند.

به این نوع برنامه‌ها **Stateless Applications** گفته می‌شود.

---

## اما همه برنامه‌ها Stateless نیستند

فرض کنید می‌خواهیم یکی از سرویس‌های زیر را اجرا کنیم:

* PostgreSQL
* MySQL
* MongoDB
* Redis
* Kafka
* RabbitMQ
* Elasticsearch
* Cassandra

در اینجا شرایط کاملاً متفاوت است.

این برنامه‌ها اطلاعات را در حافظه یا دیسک ذخیره می‌کنند.

اگر Pod از بین برود و Pod جدیدی جایگزین آن شود، نباید اطلاعات قبلی از بین برود.

---

## یک مثال ساده

فرض کنید PostgreSQL را داخل یک Deployment اجرا کرده‌ایم.

یک Pod ایجاد می‌شود.

نام آن:

```text
postgres-7bc89fd9d8-l4jvm
```

تمام داده‌های دیتابیس داخل همین Pod قرار دارد.

حالا Node دچار مشکل می‌شود.

Pod حذف می‌شود.

Deployment یک Pod جدید می‌سازد.

```text
postgres-7bc89fd9d8-k2mhd
```

این Pod جدید:

* نام متفاوتی دارد.
* IP متفاوتی دارد.
* اگر Persistent Volume نداشته باشد، اطلاعات قبلی را هم ندارد.

حتی اگر Persistent Volume داشته باشد، هنوز مشکل دیگری وجود دارد.

---

## مشکل فقط ذخیره‌سازی نیست

فرض کنید سه Replica از Kafka داریم.

آیا مهم است که Broker شماره ۱ کدام Pod باشد؟

بله.

چون Brokerها یکدیگر را با شناسه مشخص می‌شناسند.

اگر امروز Broker شماره ۱ باشد:

```text
kafka-0
```

فردا نباید تبدیل شود به:

```text
kafka-2
```

چون تمام Cluster دچار مشکل می‌شود.

---

## مشکل اصلی چیست؟

بعضی برنامه‌ها به سه ویژگی نیاز دارند:

* هویت ثابت
* ذخیره‌سازی دائمی
* ترتیب مشخص هنگام ایجاد و حذف

Deployment هیچ‌کدام از این موارد را تضمین نمی‌کند.

---

# اینجاست که StatefulSet وارد می‌شود

StatefulSet نوعی Controller در Kubernetes است که برای مدیریت برنامه‌های Stateful طراحی شده است.

هدف آن این است که هر Pod همیشه:

* نام مشخصی داشته باشد.
* هویت مشخصی داشته باشد.
* دیسک مخصوص خود را داشته باشد.
* ترتیب راه‌اندازی مشخصی داشته باشد.
* ترتیب خاموش شدن مشخصی داشته باشد.

---

# تفاوت Deployment و StatefulSet

| Deployment                                         | StatefulSet                             |
| -------------------------------------------------- | --------------------------------------- |
| مناسب برنامه‌های Stateless                         | مناسب برنامه‌های Stateful               |
| نام Podها تصادفی است                               | نام Podها ثابت است                      |
| Replicaها قابل جایگزینی هستند                      | هر Replica هویت مستقل دارد              |
| ترتیب ایجاد مهم نیست                               | ایجاد به صورت ترتیبی انجام می‌شود       |
| حذف به صورت هم‌زمان ممکن است                       | حذف به صورت معکوس و ترتیبی انجام می‌شود |
| معمولاً از یک PVC مشترک یا بدون PVC استفاده می‌شود | هر Pod یک PVC اختصاصی دارد              |

---

# هویت ثابت (Stable Identity)

اگر StatefulSet با سه Replica ایجاد کنیم:

```text
mysql-0
mysql-1
mysql-2
```

اگر Pod شماره ۱ حذف شود:

```text
mysql-1
```

Kubernetes دوباره دقیقاً همان Pod را ایجاد می‌کند.

```text
mysql-1
```

نه:

```text
mysql-8fd2
```

و نه:

```text
mysql-new
```

همیشه همان هویت حفظ می‌شود.

---

# شبکه پایدار (Stable Network Identity)

هر Pod یک DNS ثابت دارد.

مثلاً:

```text
mysql-0.mysql.default.svc.cluster.local

mysql-1.mysql.default.svc.cluster.local

mysql-2.mysql.default.svc.cluster.local
```

در نتیجه، سایر اعضای Cluster همیشه می‌دانند هر Pod چگونه قابل دسترس است.

---

# فضای ذخیره‌سازی اختصاصی

فرض کنید سه Replica داریم.

برای هر Pod یک PersistentVolumeClaim مستقل ایجاد می‌شود:

```text
mysql-0
↓

PVC-0

--------------------

mysql-1
↓

PVC-1

--------------------

mysql-2
↓

PVC-2
```

اگر Pod حذف شود، دیسک آن حذف نمی‌شود.

وقتی Pod دوباره ساخته شود، همان PVC قبلی به آن متصل خواهد شد.

---

# ترتیب ایجاد Podها

اگر سه Replica تعریف کنیم:

```text
Replica = 3
```

Kubernetes به این ترتیب عمل می‌کند:

```text
mysql-0

↓

Ready

↓

mysql-1

↓

Ready

↓

mysql-2
```

تا زمانی که `mysql-0` آماده نشود، `mysql-1` ایجاد نمی‌شود.

این رفتار برای بسیاری از پایگاه‌های داده و سیستم‌های توزیع‌شده ضروری است.

---

# ترتیب حذف

در زمان حذف نیز ترتیب برعکس است:

```text
mysql-2

↓

mysql-1

↓

mysql-0
```

این موضوع از بروز مشکلات در خوشه‌هایی مانند Kafka یا ZooKeeper جلوگیری می‌کند.

# جمع‌بندی

به خاطر داشته باشید:

* **Deployment** برای برنامه‌هایی است که Replicaها کاملاً قابل جایگزینی هستند.
* **StatefulSet** برای برنامه‌هایی است که هر Replica هویت، شبکه و فضای ذخیره‌سازی مخصوص به خود را دارد.

یک جمله که معمولاً در کلاس استفاده می‌کنم و در ذهن دانشجو می‌ماند این است:

> **اگر حذف یک Pod هیچ تفاوتی برای برنامه ایجاد نکند، احتمالاً Deployment انتخاب مناسبی است. اما اگر هر Pod شخصیت و هویت خودش را داشته باشد و حذف یا جایگزینی آن روی عملکرد سیستم اثر بگذارد، به StatefulSet نیاز دارید.**

این جمله به دانشجو کمک می‌کند به جای حفظ کردن تعریف‌ها، منطق انتخاب بین Deployment و StatefulSet را درک کند.
